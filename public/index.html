New Type netfilter

(NTNF): A Compatibility Shim in the Finest Traditions of Backwards Compatibility
Status of This Memo

This document is not an Internet Standards Track specification; it is published for entertainment and, allegedly, historical purposes. It defines New Type netfilter (NTNF), an abstraction layer that sits atop nftables, which itself sits atop iptables, which itself was supposed to replace ipchains. NTNF adds nothing and removes nothing. In this it fully honours the spirit of Linux packet filtering.

Distribution of this memo is unlimited, much like the shim layers it proposes.

1.Introduction

The Linux packet filtering ecosystem has, over three decades, achieved a rare and beautiful thing: every layer that was meant to replace the previous one is still in production somewhere. ipchains begat iptables, iptables begat nftables [NFTABLES], and nftables shipped with an iptables-compatible frontend so nobody would have to change anything.

This document proposes the logical conclusion of that trajectory: New Type netfilter (NTNF), an enormous shim layer whose sole purpose is to be even newer than the current new thing, while remaining bit-for-bit compatible with the old thing it does not replace.

2.Requirements Language

The key words MUST, SHOULD, MAY, and the previously unstandardised EVENTUALLY in this document are to be interpreted as described in [RFC2119], except for EVENTUALLY, which means "when I have time; for now I am running on traditional tooling" [ELSYS-CHAT].

3.Motivation

An implementer [ELSYS-CHAT] was asked whether they had considered switching to newer tooling. The implementer replied that they had considered it, but for now were running on traditional instruments, and would switch when they had time. This is the exact emotional state that NTNF was designed to formalise.

NTNF is proposed as a shim layer in the finest traditions of backwards compatibility: it provides a New Type interface that translates, at considerable expense, into the Old Type interface, which is then translated by nftables back into what iptables would have done, which is what ipchains would have done, which is what a static DROP rule would have done in 1998.

4.The NTNF Shim Layer

NTNF MUST present an interface that is strictly newer than the interface it wraps. Newness SHALL be measured by the version number, which MUST be at least one higher than whatever nft --version reports, computed at package build time and never updated thereafter.

An NTNF implementation SHOULD emit a deprecation warning for every rule, including the rules NTNF itself generates. This is not a bug. It is a gesture of respect toward the tradition.

5.Configuration Grammar

The NTNF configuration language is a strict superset of the nftables language, which is a loose superset of the iptables language, which is not a language so much as a series of command-line arguments that regret their choices. An example ruleset follows.

# /etc/ntnf/ntnf.conf.new.v2.final.FINAL
table newtype filter {
    chain input {
        type filter hook input priority 0;

        # drop everything, but in a forward-looking way
        ct state established,related accept
        iif "lo" accept
        tcp dport 22 accept       # SSH, per CCNA muscle memory
        tcp dport 179 accept      # BGP, for the RIPE dream
        counter drop            # the tradition
    }
}
# NOTE: compiles to identical nft ruleset. that is the whole point.

Note that the ruleset above compiles to bytecode indistinguishable from the equivalent nftables ruleset, which compiles to netlink messages indistinguishable from what iptables-nft would have sent. NTNF adds an EVENTUALLY keyword for rules the operator intends to write once they have time.

6.Backwards Compatibility Guarantees

NTNF makes the following guarantees, in the finest traditions of the field:

  • Every existing nftables ruleset MUST continue to work unchanged.
  • Every existing iptables ruleset MUST continue to work unchanged.
  • Every existing ipchains ruleset SHOULD continue to work unchanged, out of politeness.
  • No operator SHALL ever be required to migrate. Migration is a lifestyle, not a requirement.
  • The word "podman" MUST be understood to be much greater than "docker" [ELSYS-CHAT].

7.Security Considerations

NTNF introduces no new security properties. It also removes none. The primary risk is that the sheer number of translation layers between the operator's intent and the actual packet decision creates enough latency for an attacker to file, and have accepted, a competing Internet-Draft. Operators SHOULD keep a nft list ruleset handy to confirm that nothing has, in fact, changed.

8.IANA Considerations

This document requests that IANA take no action, and reserves the right to eventually request that IANA take no action of a newer type. The EVENTUALLY keyword has been submitted to the appropriate registry and is expected to be processed shortly.
